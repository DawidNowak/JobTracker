## Findings

### F1 — `"use client"` directive in shadcn select.tsx

- **Severity**: ⚠️ WARNING
- **Impact**: 🔎 MEDIUM — real tradeoff; pause to reason through it
- **Dimension**: Pattern Consistency
- **Location**: src/components/ui/select.tsx:1
- **Detail**: File begins with `"use client";`. AGENTS.md line 10 is a hard rule: *"Do not use Next.js-style directives (`use client`, `use server`)."* Plan §3.1 also says to keep generated shadcn files unedited so future `npx shadcn add` diffs cleanly. The two rules contradict each other for any shadcn primitive that ships the directive (Select today; Popover, Dropdown, Tooltip, etc. tomorrow). The directive is inert in Astro's bundler (no functional impact), so this is policy debt — but the same conflict will recur on every primitive install.
- **Fix A ⭐ Recommended**: Carve out shadcn primitives in AGENTS.md (one-line rule edit: "…except files generated by `npx shadcn`").
  - Strength: One change covers all future primitive installs; preserves diff-merge for upstream updates.
  - Tradeoff: Slightly weakens AGENTS.md's absolute-rule posture.
  - Confidence: HIGH — directive is inert in Astro's pipeline; rule is hygiene, not correctness.
  - Blind spot: None significant.
- **Fix B**: Strip the directive from `select.tsx` and document the deviation.
  - Strength: Honors the literal AGENTS.md rule today.
  - Tradeoff: Has to be redone every `npx shadcn add` reinstall; silent regression risk.
  - Confidence: MED — depends on planned reinstall cadence.
  - Blind spot: Other shadcn primitives may also carry the directive (Select is just the first to land).
- **Decision**: FIXED via Fix A (AGENTS.md line 10 amended to exempt `src/components/ui/` shadcn-generated files).

### F3 — Mixed Polish/English error strings in API route

- **Severity**: OBSERVATION
- **Impact**: 🏃 LOW
- **Dimension**: Pattern Consistency
- **Location**: src/pages/api/applications/index.ts:33, 40, 50, 59
- **Detail**: Error envelopes mix languages: `"Unauthorized."` (401), `"Invalid JSON body."` (400), `"Supabase is not configured."` (500) are English; `"Nie udało się zapisać aplikacji."` (500-catch) is Polish. The dialog only renders the Polish catch-all message; the others surface to curl / devtools. Plan didn't prescribe a language convention — pick one now while there's one endpoint, not five.
- **Fix**: Either translate the three English strings to Polish (`"Brak autoryzacji."`, `"Nieprawidłowe ciało żądania JSON."`, `"Supabase nie jest skonfigurowany."`) OR document an English-for-system / Polish-for-user split in the plan / AGENTS.md. S-03/S-04 will inherit whichever convention is chosen.
- **Decision**: FIXED — translated to Polish (`"Brak autoryzacji."`, `"Nieprawidłowe żądanie"`, `"Supabase nie jest skonfigurowany."`); uniform Polish-for-user convention.

### F4 — Unsafe widening cast on `addableStatuses.includes`

- **Severity**: OBSERVATION
- **Impact**: 🏃 LOW
- **Dimension**: Architecture
- **Location**: src/components/board/KanbanBoard.astro:14, 24
- **Detail**: `addableStatuses.includes(status)` does not narrow `status` from the full `ApplicationStatus` union to the two-status subset, so the `targetStatus` prop binding uses `status as "Interesujące" | "Zaaplikowano"`. The cast is currently safe (`addableStatuses` and the cast target are the same literal union by construction), but the two literals can drift independently if either is edited without the other.
- **Fix**: Replace with an inline type-guard:
  ```astro
  {(status === "Interesujące" || status === "Zaaplikowano") && (
    <AddApplicationDialog client:load targetStatus={status} slot="header-action" />
  )}
  ```
  drops the cast and keeps the two lists in sync at the type level.
- **Decision**: FIXED (replaced `addableStatuses.includes` + cast with an inline literal type-guard; dropped the unused `addableStatuses` constant).

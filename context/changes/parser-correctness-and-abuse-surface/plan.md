# Parser correctness + abuse surface — Implementation Plan

## Overview

Land the test-plan §3 Phase 2 wedge: lock parser correctness against silent field drift (Risk #1) and prove `recognize()` gates outbound `fetch` on the SSRF surface (Risk #4). Add a Workers-runtime Vitest pool so parser unit tests can call `HTMLRewriter` natively, capture a 6-fixture suite of real portal HTML, ship a `recognize()` URL-classifier table + endpoint-level SSRF assertion, and bundle three small defence-in-depth hardening changes — each paired with its own regression test.

## Current State Analysis

- `src/lib/parsers/recognize.ts` is the single SSRF boundary: 36 lines, post-F3 narrowed to three exact LinkedIn hosts (`linkedin.com`, `www.linkedin.com`, `pl.linkedin.com`) plus bare `justjoin.it`. Allowlist branches at `recognize.ts:18` and `recognize.ts:30`; protocol gate at `recognize.ts:14`.
- `src/pages/api/applications/parse.ts:28-43` owns the status-classification matrix (`ok` / `partial` / `empty` / `unsupported` / `fetch_failed`) that `AddApplicationDialog.tsx` depends on for its banner-vs-prefill behaviour.
- `src/lib/parsers/linkedin.ts` + `src/lib/parsers/justjoinit.ts` use `HTMLRewriter` (workerd-only global, declared ambient at `src/lib/parsers/html-rewriter.d.ts:22`). Neither sets `redirect:` on its outbound `fetch` (Workers default is `follow`); JJIT does not wrap `slug` in `encodeURIComponent` (currently safe — `recognize.ts:31` regex constrains slug to `[a-z0-9-]+`); the parse endpoint does not re-validate `recognized.jobId` / `recognized.slug` between `recognize()` and the parser call.
- Two history-proven silent-drift bugs shipped from this surface: LinkedIn description tag-collapse + "Show more" leak (commit `142dc93`) and JJIT `workplace_type` → `workplaceType` marker-key rename (commit `2b9e722`). Both invisible to status checks; only caught by humans noticing wrong values.
- Test infrastructure from Phase 1 + 3 is reusable: `tests/helpers/users.ts`, `tests/helpers/cookies.ts`, `tests/global-setup.ts:73-99` (spawns `astro dev` and exports `TEST_BASE_URL`). `vitest.config.ts` is single-project, `environment: "node"`. There are zero parser tests and zero captured HTML fixtures in the repo today.
- `package.json` already has `vitest@^3.2.0`, `wrangler@^4.93.0`, `@astrojs/cloudflare@^13.5.0`. No `@cloudflare/vitest-pool-workers` yet.

## Desired End State

- A Workers-runtime Vitest pool runs parser unit tests in-process against `HTMLRewriter` without any cross-process stubbing; the existing node pool continues to run integration + HTTP suites.
- `tests/fixtures/parsers/{linkedin,justjoinit}/` holds three real HTML payloads per portal (happy / field-missing / corrupted), each with hand-typed oracle assertions in a sibling test file. A `tests/fixtures/parsers/README.md` documents the capture procedure and the capture-date + source-URL per file.
- `tests/unit/parsers/recognize.test.ts` is a table-driven classifier test covering both allowed branches and bypass-attempt rows from the research §B table.
- `tests/http/parse-applications.test.ts` exercises `POST /api/applications/parse` against the running `astro dev`, asserts `fetch` is never called for disallowed inputs (via the in-test `fetch` stub helper), and the `unsupported` / `fetch_failed` / `ok` envelope shapes match the contract `AddApplicationDialog.tsx` reads.
- Three defence-in-depth changes have landed in `src/lib/parsers/`: `redirect: "manual"` on both outbound fetches, `encodeURIComponent` on the JJIT slug, an explicit input regex re-check inside each parser. Each has a regression test that reds if reverted.
- `npm test` passes locally and is wired (`vitest projects: ["./tests/projects/node.ts", "./tests/projects/workers.ts"]` or equivalent) such that both pools run in one invocation.
- `tests/README.md` describes the two-pool layout; `context/foundation/test-plan.md` §4 "API mocking" row and §6.4 "Adding a test for the parser layer" are filled in (TBD removed).

### Key Discoveries

- `HTMLRewriter` lives in `src/lib/parsers/html-rewriter.d.ts:22` as ambient — pool-workers wiring needs to import the workerd type, not redeclare it.
- `tests/global-setup.ts:73-99` already swaps `.dev.vars` and spawns `astro dev`. Both pools should share this global setup; per-pool `setupFiles` may differ (the workers pool doesn't need the Supabase URL guard from `tests/setup.ts:1`).
- `parse.ts:65-79` calls `recognize()` exactly once and passes the extracted id/slug into the parser. The parser-side re-check in Phase 4 sits inside the parser entry function, not in `parse.ts`, so the boundary stays single-gated at the route while parsers fail-closed on inconsistent input.
- The research-recommended oracle pattern (hand-type expected values from the visible job page) is the only honest signal for Risk #1. Tests must not be generated by running the parser once and freezing the output.

## What We're NOT Doing

- **Scheduled HTML drift canary against live URLs.** Deferred to test-plan §3 Phase 4 (CI quality-gate wiring) where the GitHub Actions cron + secret + LinkedIn-HTTP-999 handling already belong.
- **Snapshot-style tests against captured fixtures.** `toMatchSnapshot()` over parser output is the Risk #1 anti-pattern; all assertions are hand-typed oracles read from the visible job page at capture time.
- **End-to-end "parse → save → re-read the row" flow.** Out of scope; Phase 3 (`tests/http/post-applications.test.ts`) already exercises the row-level persistence path. Phase 2 stops at the endpoint envelope.
- **Live-LinkedIn fetch in CI.** Workers egress IPs hit the HTTP 999 block 30-60% of the time per `parser-driven-add/research.md:84-89`. Fixtures sidestep this entirely.
- **Refactoring `parse.ts` or `AddApplicationDialog.tsx`.** Their behaviour is the system under test, not the target of change. Only the three parser-internal hardening lines are touched.
- **Coverage thresholds.** Per test-plan §7, signal-on-top-risks beats a coverage number.
- **`@cloudflare/vitest-pool-workers` for non-parser tests.** Integration and HTTP suites stay on the node pool. Adding pool-workers everywhere would force every Supabase test into workerd's `fetch` and re-open the cross-runtime can of worms.

## Implementation Approach

Four phases, ordered cheapest-signal-first so the most valuable assertions land before the riskiest setup work.

1. **Workers pool infrastructure** comes first because both #1 and #4 parser-level tests depend on it. This phase is risk-front-loaded: if pool-workers and `@astrojs/cloudflare` v13.5 don't cohabit, we discover it before any test is written.
2. **Risk #4** lands second because the `recognize()` table is pure-function unit testing — no fixtures, no dev server. The endpoint SSRF test then reuses the existing `astro dev` scaffold from Phase 3 and only needs a `fetch` stub helper.
3. **Risk #1** comes third because it carries the highest setup cost (HTML capture per portal) and is the most brittle to drift. Landing it after the runner is proven keeps the failure surface local.
4. **Hardening regressions** land last, each paired with its own targeted test so the hardening line and its lock-in commit together.

The Progress section at the bottom enumerates the verification checklist for every phase.

## Critical Implementation Details

- **The oracle problem (Phase 3).** For each captured fixture, the test author opens the live job page in a browser at capture time and types the expected `position`, `company`, salient lines from `description`, `salary`, and `work_mode` into the test file. Assertions read from the page, not from the parser's current output. A re-capture of a fixture later requires a re-read of the page — never a re-run of the parser to "refresh" expectations.
- **Pool-vs-pool boundary.** Parser unit tests run under workers pool (need `HTMLRewriter`); `recognize()` classifier test runs under node pool (pure function, no workerd dependency); HTTP endpoint test runs under node pool (uses `tests/global-setup.ts` `astro dev` spawn). The vitest `projects` config splits these by `include:` glob — `tests/unit/parsers/recognize.test.ts` is node, `tests/unit/parsers/*.{linkedin,justjoinit}.test.ts` are workers, `tests/http/**` is node.

## Phase 2: Risk #4 — `recognize()` classifier + endpoint SSRF test

### Overview

Lock the SSRF boundary with two complementary tests: a pure-function table over `recognize()` covering the post-F3 allowlist + every bypass-attempt shape in research §B, and an endpoint-level assertion that `fetch` is never called when the input is rejected by `recognize()`.

### Changes Required

#### 1. `fetch` stub helper

**File**: `tests/helpers/fetch.ts`

**Intent**: A `withFetchStub(handler, fn)` wrapper around `vi.stubGlobal("fetch", ...)` + `vi.unstubAllGlobals()` so tests can install a per-test stub without leaking. Used by phase 2 endpoint test and phase 4 redirect test.

**Contract**: Exports `withFetchStub(handler: (req: Request) => Response | Promise<Response>, fn: () => Promise<void>): Promise<void>`. Internally restores the original `globalThis.fetch` in `finally`. No DI seam needed — the parsers call the global directly.

#### 2. `recognize()` classifier table

**File**: `tests/unit/parsers/recognize.test.ts`

**Intent**: One `describe` with a parametrized table covering every row from research §B. Each row: `{ input, expected: null | { kind, jobId? | slug? } }`. The table includes the three LinkedIn host equalities, JJIT slug shape, bypass attempts (trailing dot, mixed case, userinfo, IDN look-alikes, suffix-confusion hosts, JJIT `www.` rejection, non-http protocols, etc.), and the post-F3 narrowing regression rows (e.g. `evil.linkedin.com` must return `null`).

**Contract**: `it.each(table)` over a const array. Each row asserts `recognize(input)` equals `expected` by deep equality. No mocks, no stubs — pure function under test. Runs in node pool.

#### 3. Endpoint SSRF test

**File**: `tests/http/parse-applications.test.ts`

**Intent**: Drive `POST /api/applications/parse` over HTTP against `astro dev`, with `fetch` stubbed at the dev-server level. For disallowed inputs (non-portal URLs, internal IPs, `javascript:`, `evil.linkedin.com`), assert response is `{ status: "unsupported", message: ... }` with HTTP 200 and **zero outbound `fetch` calls were made beyond the test's own dev-server request**. For allowed inputs, assert `fetch` was called with exactly the expected URL literal.

**Contract**: Test runs under node pool, follows the pattern in `tests/http/post-applications.test.ts:8-44` for auth/session setup. Stubs `fetch` via `withFetchStub` from `tests/helpers/fetch.ts`. **Implementation note**: stubbing `fetch` in the test process does not affect the dev-server process; the assertion must read the envelope shape returned by the dev-server and infer "fetch not called" from the `unsupported` envelope (which short-circuits at `parse.ts:65-72` before the parser runs). The "zero outbound" claim is structurally guaranteed by the `unsupported` short-circuit, so the assertion is `expect(json.status).toBe("unsupported")` — the SSRF guarantee is encoded in the envelope contract.

### Success Criteria

#### Automated Verification

- `npm test` passes both new files in the node pool
- `recognize()` table covers at minimum: 3 LinkedIn host equalities (positive), JJIT slug positive, JJIT `www.` negative, trailing-dot host, mixed-case host, userinfo bypass, IDN punycode, suffix-confusion host (`evil-linkedin.com`, `attacker.com.linkedin.com`), `javascript:` / `data:` / `file:` protocols, JJIT path with extra segments
- Endpoint test covers at minimum: unauthenticated 401, valid LinkedIn URL returns one of `ok`/`partial`/`fetch_failed`, disallowed URL returns `unsupported`, malformed body returns 400, Zod-fail returns 422
- `npm run typecheck` passes
- `npm run lint` passes

#### Manual Verification

- Mutating `recognize.ts:18` back to the pre-F3 suffix-match (e.g. `host.endsWith(".linkedin.com")`) reds at least one row in the classifier table (proves the F3 regression guard is real)
- Mutating `recognize.ts:14` to accept `javascript:` reds the protocol row

**Implementation Note**: After phase 2 lands, pause for manual confirmation that the `recognize()` table and endpoint test red appropriately under intentional regressions. Then proceed.

## Phase 3: Risk #1 — fixture suite + parser tests

### Overview

Capture three real HTML payloads per portal, store them under `tests/fixtures/parsers/{linkedin,justjoinit}/`, write hand-typed oracle assertions in per-portal test files under the workers pool, and add a synthetic-input unit test on `resolveStatus` to lock the per-portal status-classification matrix.

### Changes Required

#### 1. Fixture directory + capture README

**Files**: `tests/fixtures/parsers/README.md`, `tests/fixtures/parsers/linkedin/{happy,missing-salary,corrupted}.html`, `tests/fixtures/parsers/justjoinit/{happy,missing-salary,corrupted}.html`

**Intent**: Three captured payloads per portal. Happy = a real recent posting with all expected fields. Field-missing = a real posting where one optional field (salary for both portals) is genuinely absent on the page. Corrupted = a happy fixture with one critical element removed by hand (LinkedIn: strip `.topcard__title`; JJIT: strip the `"workplaceType"` marker from the Flight payload) to trigger `fetch_failed`.

**Contract**: README documents the capture procedure: `curl -A "<UA-string>" "<url>" > <portal>/<name>.html`, the UA string used, the source URL captured, and the capture date for each file. README also names the visible job page used as oracle for the happy + missing fixtures (so future maintainers can re-derive expected values without running the parser).

#### 2. LinkedIn parser tests

**File**: `tests/unit/parsers/linkedin.test.ts`

**Intent**: Three `it` blocks, one per fixture. Each reads its fixture, stubs `fetch` to return `new Response(html, { status: 200 })`, calls `parseLinkedIn(jobId)`, and asserts against hand-typed oracle values. Happy: all five fields populated with expected literals. Field-missing: `salary` is `undefined` (not `""`, not a default). Corrupted: the call throws — assertion via `expect(...).rejects.toThrow()`.

**Contract**: Runs under workers pool (needs `HTMLRewriter`). Uses `withFetchStub` from `tests/helpers/fetch.ts`. Oracle values are hardcoded strings in the test file — never derived by calling the parser. Each fixture's oracle block has a comment naming the source URL + capture date.

#### 3. JJIT parser tests

**File**: `tests/unit/parsers/justjoinit.test.ts`

**Intent**: Mirror structure of the LinkedIn test. Happy: `position`, `company`, `description` (starts with `Wymagane umiejętności:` from the skills prepend), `salary` (formatted multi-row), `work_mode`. Field-missing: `salary` is `undefined`. Corrupted: a Flight payload with the `"workplaceType"` marker removed throws → `fetch_failed`.

**Contract**: Runs under workers pool. Same `withFetchStub` pattern. Oracle values hardcoded with source-URL + capture-date comments.

#### 4. `resolveStatus` unit test

**File**: `tests/unit/parsers/resolve-status.test.ts`

**Intent**: Pure-function table over `resolveStatus` from `parse.ts:37-43`. For each portal kind (`linkedin`, `jjit`), iterate the matrix: all expected populated → `ok`; any expected missing → `partial`; nothing populated → `empty`. This is the regression guard against a future change that treats "any field populated" as `ok` (the test-plan §2 anti-pattern). The function is not exported today — phase 3 either exports it from `parse.ts` or extracts it to `src/lib/parsers/status.ts` and re-imports in `parse.ts`. Prefer extraction: keeps `parse.ts` route-shape thin.

**Contract**: New file `src/lib/parsers/status.ts` exports `EXPECTED_KEYS` (moved from `parse.ts:28-31`), `countDefined`, and `resolveStatus`. `parse.ts` imports them. Test runs in node pool (pure function, no `HTMLRewriter`). Table assertions are deep equality.

#### 5. Cookbook + tracker update

**Files**: `tests/README.md`, `context/foundation/test-plan.md`

**Intent**: Replace the "TBD — see §3 Phase 2" placeholders. `tests/README.md` gains a "Parser fixtures" subsection pointing at `tests/fixtures/parsers/README.md`. `context/foundation/test-plan.md` §6.4 fills in with the fixture-capture + oracle pattern. §3 Phase 2 status updates to `implementing` (or `complete` after phase 4 lands), and the `Change folder` column gets the path.

**Contract**: Documentation only; no behaviour change.

## Phase 4: Defence-in-depth hardening + regression tests

### Overview

Three small parser-internal changes, each shipped with a regression test that reds if the change is reverted. None of these is currently exploitable; all narrow the blast radius of a future `recognize()` regression.

### Changes Required

#### 1. `redirect: "manual"` on both outbound fetches

**Files**: `src/lib/parsers/linkedin.ts`, `src/lib/parsers/justjoinit.ts`

**Intent**: Add `redirect: "manual"` to the existing `fetch` options object in each parser. When the upstream returns 3xx, the parser sees a synthetic empty response with status 0 (or status 3xx depending on the runtime); detect and throw, collapsing to `fetch_failed`.

**Contract**: `linkedin.ts:32-39` and `justjoinit.ts:222-229` gain `redirect: "manual"`. Each parser inspects `response.status` after the `fetch` call and throws if `response.status >= 300 && response.status < 400` (or `response.type === "opaqueredirect"`, depending on what workerd actually surfaces — verify by stubbing a 302 in the test).

#### 2. `encodeURIComponent` on JJIT slug

**File**: `src/lib/parsers/justjoinit.ts`

**Intent**: Wrap `slug` in `encodeURIComponent` at the URL interpolation site (`justjoinit.ts:222`). Symmetry with `linkedin.ts:32`. Belt-and-suspenders against any future loosening of `recognize.ts:31`.

**Contract**: `https://justjoin.it/job-offer/${slug}` becomes `https://justjoin.it/job-offer/${encodeURIComponent(slug)}`. Behaviour unchanged for currently-valid slugs (the regex `[a-z0-9-]+` produces only characters that `encodeURIComponent` leaves untouched).

#### 3. Parser-side input re-check

**Files**: `src/lib/parsers/linkedin.ts`, `src/lib/parsers/justjoinit.ts`

**Intent**: At the top of `parseLinkedIn(jobId)` and `parseJustJoinIT(slug)`, re-validate the argument against the same regex `recognize()` uses (`/^\d{8,}$/` for LinkedIn, `/^[a-z0-9-]+$/` for JJIT). Throw immediately on failure → `fetch_failed` envelope. Defence-in-depth against a future `recognize()` regression or a future caller that bypasses `recognize()`.

**Contract**: One `if (!regex.test(arg)) throw new Error("invalid <arg>")` at the top of each parser entry function. The regex literal is co-located with the parser (not re-imported from `recognize.ts`) so the two layers can drift apart only by deliberate edit.

#### 4. Regression tests for each hardening change

**Files**: `tests/unit/parsers/linkedin.test.ts`, `tests/unit/parsers/justjoinit.test.ts` (extend existing files from phase 3)

**Intent**: Three new `it` blocks per portal:
- Redirect test: stub `fetch` to return `new Response(null, { status: 302, headers: { Location: "https://example.com/" } })`; assert parser throws.
- Input re-check test: call `parseLinkedIn("not-a-jobid")` / `parseJustJoinIT("Not-A-Slug!")` directly; assert throws **without** any `fetch` call (`expect(fetch).not.toHaveBeenCalled()`).
- (JJIT only) Slug encoding test: call `parseJustJoinIT("valid-slug-123")` with the regex check disabled (by passing in a slug containing a `/` via a temporary direct call after mutating the regex to allow it — alternatively, assert the URL passed to `fetch` via a stub spy contains `encodeURIComponent`'d form for an edge-case input). Simpler approach: assert `fetch.mock.calls[0][0]` matches `https://justjoin.it/job-offer/valid-slug-123` for the happy path AND that an explicit `parseJustJoinIT("a%2Fb")` would not interpolate raw `/` (verify via stub spy).

**Contract**: Each new test runs under workers pool. Uses `withFetchStub`. The redirect and input-re-check tests are the regression guards for hardening changes 1 and 3; the encoding test guards change 2.

#### 5. Tracker close-out

**File**: `context/foundation/test-plan.md`, `context/changes/parser-correctness-and-abuse-surface/change.md`

**Intent**: Flip `change.md` `status: planned` → `status: complete` once Progress is fully `[x]`. Update test-plan §3 Phase 2 row `Status` to `complete`. Add a one-line note in §5 "Parser HTML fixture suite" row confirming the gate is now wired locally + CI.

**Contract**: Documentation update only.

### Success Criteria

#### Automated Verification

- `npm test` passes the new regression tests in both parser test files
- Reverting any single hardening change reds at least one test
- `npm run typecheck` passes
- `npm run lint` passes
- `npm run build` passes (parser changes ship cleanly through the Astro build)

#### Manual Verification

- A `wrangler dev` smoke test against a real LinkedIn URL still parses correctly (the `redirect: "manual"` change does not red on a current LinkedIn response — `linkedin.com/jobs-guest/...` returns 200 directly in normal flow)
- A `wrangler dev` smoke test against a real JJIT URL still parses correctly (the slug encoding and input re-check do not red on a current JJIT response)
- Both portals' parse buttons still pre-fill the dialog as before when used through the UI

## Testing Strategy

### Unit Tests

- `recognize()` URL-classifier table — pure function, node pool, covers post-F3 allowlist + every bypass shape from research §B
- `resolveStatus` matrix — pure function, node pool, covers per-portal `EXPECTED_KEYS` against synthetic `ParseResult` inputs
- Parser fixture tests — workers pool, three captured HTML payloads per portal asserted against hand-typed oracle values

### Integration Tests

- `POST /api/applications/parse` endpoint — node pool via `astro dev`, asserts envelope shapes for unauthenticated / disallowed / allowed / malformed inputs

### Manual Testing Steps

1. After phase 1: `npm test` reports two project names in the reporter (`node` and `workers`). The smoke test passes.
2. After phase 2: deliberately mutate `recognize.ts:18` to a suffix-match; confirm a classifier-table row reds.
3. After phase 3: open the captured LinkedIn fixture's source URL in a browser; visually confirm the values the test asserts match what the page shows.
4. After phase 4: `wrangler dev`, hit `/board/add` in the UI, paste a real LinkedIn URL and a real JJIT URL; confirm the dialog pre-fills correctly (no regression from the hardening changes).

## Performance Considerations

- Workers pool startup adds ~1-2s to `npm test` cold-runs. Acceptable; the suite is small.
- Fixture files commit ~50-200 KB each (LinkedIn guest fragment is small; JJIT page is larger due to the Flight payload). Six fixtures ≈ ~600 KB total. Acceptable.

## Migration Notes

None. No schema changes; no data changes; no user-visible behaviour change beyond the redirect-handling tightening (rejects 3xx upstream responses, which neither portal currently emits on the parsed endpoints).
